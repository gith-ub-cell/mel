# 实践映射手册

> "如果有了正确的理论，只是把它空谈一阵，束之高阁，并不实行，那末，这种理论再好也是没有意义的。" —— 毛泽东《实践论》

本手册将辩证唯物论、认识论、唯物史观的核心原理映射为具体的诊断和行动工具。

---

## 一、科研场景

### 实验结果波动大

**诊断流程**：

1. **区分现象和本质**：
   - 现象：每次跑出来的结果不一样
   - 可能本质：数据问题？随机性问题？代码 Bug？评估指标定义不清？
2. **检查物质条件**（先盘实物）：
   - 数据集版本是否一致？预处理是否有随机性？
   - 随机种子是否固定？框架版本是否一致？
   - 硬件（GPU 型号、CUDA 版本）是否一致？
3. **给出最小复现实验**：
   - 固定所有变量，只改变一个变量
   - 目标是让波动可复现——能稳定复现的 Bug 就解决了一半

**矛盾分析**：
- 主要矛盾通常是：实验的可复现性 vs 实验的探索性
- 矛盾的主要方面：如果还在探索期，不能为了可复现性牺牲试错速度
- 如果到了收敛期，可复现性必须成为主要方面

### 路线争议大

**诊断步骤**：
1. 将争论各方的主张写下来（不要口头争论）
2. 明确分歧是在哪个层面：
   - 数据层面（用什么数据训练/评估）
   - 方法层面（用什么算法/架构）
   - 评价层面（用什么指标判断好坏）
   - 资源层面（时间和算力够不够试所有方案）
3. 设计一个最小对比实验来裁决

**经典映射**：
> "真理的标准只能是社会的实践。" —— 毛泽东《实践论》

与其争论"谁的方案更好"，不如设计一个实验让数据说话。

## 二、工程与调试场景

### 线上问题反复出现

**分类诊断框架**：

```
现象（客观描述，不推测）：
  "服务每隔 2 小时 OOM，重启后恢复"

可能本质（从现象深入到机制）：
  □ 内存泄漏（对象创建后未释放）
  □ 流量尖峰（定时任务集中触发）
  □ 配置限制（heap 设置低于实际需求）
  □ 第三方依赖（某个库的已知 bug）

主要矛盾判断（哪一层？）：
  □ 代码层（逻辑错误、资源未释放）
  □ 依赖层（第三方库的 bug 或不当使用）
  □ 配置层（参数设置不符合实际负载）
  □ 流程层（部署、回滚、监控流程的缺陷）
  □ 资源层（硬件、网络、配额不足）

实践方案（最小可验证的干预）：
  1. 先加监控：在 OOM 前 5 分钟 dump heap
  2. 复现环境：用相同流量模式在 staging 复现
  3. 逐个排除：每次改变一个变量

验证标准（可观测的判据）：
  - 修复后连续运行 48 小时无 OOM
  - heap 使用率稳定在 X% 以内
```

### 局部性能优化

**系统联系检查清单**：

```
□ 可读性影响：优化后的代码别人还能看懂吗？
□ 维护成本：引入新依赖了吗？增加了什么复杂度？
□ 对其他模块的影响：改了共享的数据结构吗？
□ 对测试的影响：原有的测试还能通过吗？需要新增测试吗？
□ 是否在优化非瓶颈：profile 数据确实证明这里是瓶颈吗？
□ 优化的量级：优化 1% 值得付出的代价吗？
```

**经典警示**：
> "不要小看联系。局部最优常常伤害整体。优化了数据库查询但增加了网络往返，整体反而变慢了。"

## 三、项目推进场景

### 项目推进不动

**不要立刻说"人不行"**，按照以下顺序诊断：

```
第一层：物质条件
  □ 需求是否稳定？还是在不断变化？
  □ 资源是否到位？（人、时间、工具、权限）
  □ 上下游依赖是否通畅？（等设计、等接口、等审批）

第二层：目标与阶段
  □ 目标是否清晰且可衡量？
  □ 所有人对目标的理解是否一致？
  □ 当前阶段的主要任务是否正确？
    - 探索期：过度要求规范和文档 → 拖死
    - 成长期：缺乏规范和自动化 → 乱死

第三层：结构与激励
  □ 分工是否清晰？有没有重叠或空白？
  □ 做得好有没有正向反馈？做得差有没有代价？
  □ 决策权是否与实际能力匹配？

第四层：矛盾诊断
  主要矛盾：
  矛盾的主要方面：
  次要矛盾：
```

### 需求反复变更

**现象与本质分离**：

| 现象 | 可能本质 |
|------|----------|
| 需求频繁变更 | 真正的用户需求还未搞清楚（还处在探索期） |
| 需求频繁变更 | 需求方和实现方之间缺乏验收标准 |
| 需求频繁变更 | 决策链路过长，需求被层层转达时变形 |
| 需求频繁变更 | 用户看到了初版才发现自己真正想要什么（正常的认识过程） |

**实践方案**：
- 如果是探索期正常现象 → 缩短迭代周期，快速试错
- 如果是验收标准缺失 → 先停下来定义"什么叫做完"
- 如果是信息传递变形 → 让需求方直接和实现方沟通

## 四、组织与团队场景

### 团队士气问题

**唯物史观诊断**（先看存在，再看意识）：

```
1. 实际的工作条件：
   - 加班是否常态化？是任务多还是效率低？
   - 工具是否顺手？构建时间是否过长？
   - 工作环境是否有不合理的干扰？

2. 实际的分工与回报：
   - 能力强的人是否承担了更多但没有相应回报？
   - 混日子的人是否和拼命干的人拿一样的评价？
   - Bug 修复和技术债务清理是否被认可？

3. 实际的成长空间：
   - 成员是否在重复同样的工作？
   - 是否有时间和资源学习？
   - 技术决策是否总是由同一小群人做出？

4. 制度和文化的匹配度：
   - 季度考核是否和实际工作节奏匹配？（长期项目被迫拆成短期交付）
   - "鼓励创新"的文化下，失败是否真的不被惩罚？
```

### 流程改革推动

**"两个绝不会"的应用**：

> "无论哪一个社会形态，在它所能容纳的全部生产力发挥出来以前，是决不会灭亡的；而新的更高的生产关系，在它的物质存在条件在旧社会的胎胞里成熟以前，是决不会出现的。" —— 马克思《政治经济学批判·序言》

- 先证明旧流程的潜力已经耗尽（用数据，不要用感受）
- 先让新流程的条件在旧框架内生长（试点，不要全面铺开）
- 改革不是推翻，是扬弃——保留旧流程中合理的部分

## 五、产品与开源场景

### 开源项目没人用

**系统联系诊断**：

```
物质条件（项目本身）：
  □ README：第一屏能否让目标用户知道"这解决什么问题"？
  □ 安装：能否一条命令安装成功？时间 < 5 分钟？
  □ Demo：有没有不用安装就能看到效果的途径？
  □ 文档：有没有从零到第一个结果的指南？

传播条件（怎么被发现）：
  □ GitHub topics 和 description 是否包含用户会搜的关键词？
  □ 有没有在目标用户聚集的渠道出现过？（论坛、社区、周报）
  □ 有没有对比文章说明和竞品的区别？

信任条件（怎么被信赖）：
  □ 有没有 CI badge？测试覆盖率？
  □ 有没有 license？
  □ 有没有 release notes？
  □ 有没有 issue template？

反馈条件（怎么留住用户）：
  □ 有没有快速反馈渠道？
  □ issue 有没有被及时回复？
  □ 有没有贡献指南？
```

**矛盾分析**：
- 主要矛盾通常是："项目有能力解决某问题" vs "目标用户无法感知到这个能力"
- 实践方案：优先解决感知问题（README、Demo、对比文章），再优化能力

### 产品定位模糊

**"谁是你的用户"——阶级分析方法**：

1. 目标用户群体是谁？
2. 他们当前"生产关系"中的位置是什么？（工作中的痛点在哪里？）
3. 你的产品改变了他们什么？（效率？收益？体验？）
4. 谁会因为你的产品受益？谁可能受损？

**经典映射**：
> "谁是我们的敌人？谁是我们的朋友？这个问题是革命的首要问题。" —— 毛泽东《中国社会各阶级的分析》

产品语境下的转译：**谁是你的核心用户？谁不是你的用户？这个问题是产品定位的首要问题。**

## 八、阶段判断指南

### 五阶段模型

| 阶段 | 核心特征 | 主要任务 | 关键指标 | 主要风险 | 不应做的事 |
|------|----------|----------|----------|----------|-----------|
| **探索期** | 方向不明，方案多变 | 快速试错，找 PMF | 用户反馈频率 | 过早规模化 | 追求完美、建重流程 |
| **验证期** | 方向初定，待验证 | 做 MVP，测核心假设 | 核心假设通过率 | 虚假验证 | 加功能、追求用户量 |
| **成长期** | 方向明确，规模扩张 | 标准化、自动化 | 效率/质量平衡点 | 技术债务积累 | 忽视基础设施 |
| **成熟期** | 格局稳定，增长放缓 | 优化、降本、防守 | 边际收益 | 路径依赖、创新停滞 | 大破大立 |
| **转型期** | 旧模式失效 | 扬弃旧模式、探索新方向 | 新方向可行性 | 时机判断错误 | 同时维持新旧两条线过久 |

### 阶段判断流程

```
1. 当前项目/产品处于什么阶段？（对照上表特征）
2. 当前阶段的主要任务我们是否在做？
   - 如果探索期在做成熟期的事 → 过度工程
   - 如果成熟期在做探索期的事 → 缺乏聚焦
3. 下一个阶段的信号出现了吗？
4. 阶段转换的条件具备了吗？（"两个绝不会"）
```

## 九、快速参考卡片

### 当…时，优先检查…

| 症状 | 优先检查 | 使用工具 |
|------|----------|----------|
| 反复修反复坏 | 现象 vs 本质 | 矛盾分析表 |
| 推进不动 | 物质条件 → 阶段判断 | 项目诊断四层 |
| 团队士气低 | 激励机制 → 分工结构 | 唯物史观诊断 |
| 需求频繁变 | 目标对齐 → 验收标准 | 现象本质分离 |
| 没人用 | README → 安装体验 → 传播 | OSS 诊断清单 |
| 性能问题 | 是否瓶颈 → 系统影响 | 系统联系检查 |
| 方案争议 | 分歧层面 → 最小对比实验 | 实践检验模板 |
| 要做重大决策 | 客观条件 → 阶段判断 → 历史上类似决策的结果 | 历史分析四步法 |

### 一句话原则

- **辩证唯物论**：先看事实，再看联系，再看发展，再看矛盾
- **认识论**：不争论，做实验
- **唯物史观**：先看存在（分工、资源、奖惩），再看意识（观念、文化、口号）
- **矛盾论**：所有问题中只有一个主要矛盾，找出它，抓住它的主要方面
- **实践论**：每个判断附一个验证标准，每个行动附一个反馈回路
